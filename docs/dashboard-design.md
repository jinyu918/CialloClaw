

# 仪表盘展示文档

# 文档定位

> **文档定位：** 本稿只说明仪表盘要承载哪些内容、用户在不同状态下看到什么，以及如何进入、接管和收束交互；不展开具体技术实现。

# 1. 文档范围与设计原则

仪表盘不是单纯的功能入口页，它应该是 Agent 在桌面端的统一观察面、协作面和控制面。用户打开仪表盘后，不是去“找功能”，而是先快速判断当前最值得关注的事情，再决定是否进入某个模块接管处理。

因此，仪表盘在内容组织上要先回答四个最实际的问题：

- Agent 现在在做什么，已经做到哪一步，下一步是什么。
- 哪些信息只是提醒，哪些需要我立刻确认，哪些可以稍后处理。
- 最近生成了什么成果、发生了什么风险、当前是否存在可恢复点。
- 记忆、系统状态和任务巡检如何共同影响当前建议。

整体交互建议遵循以下原则：

- **可追溯：** 用户随时能回看 Agent 做过什么、为什么这样建议、结果落到了哪里。
- **可接管：** 涉及发送、改写、执行、越界等动作时，用户必须能显式接管。
- **可恢复：** 一旦中断或出错，用户要能看见停在哪一步、还能否继续、是否能回到安全状态。
- **统一感：** 首页总览、模块详情、召唤提示、快捷键应共享同一套状态语言，避免一个模块一种说法。

# 2. 信息架构总览

• **首页 / 意识场：**承载当前焦点、候选事项、任务巡检与召唤摘要、语音入口和全局状态，是用户的第一落点。

 首页采用分层的“意识场”式总览：

- **第一层：** 焦点对象。用于承载当前最重要、最值得立刻处理的任务，是首页默认主角。
- **第二层：** 候选对象。用于承载近期提醒、便签巡检、镜子洞察等次一级信息，提醒用户“还有这些值得看”。
- **第三层：** 背景摘要。用于承载任务巡检摘要、镜子提醒、全局状态与低打扰召唤信息，默认不抢焦点，只维持在场感。

• **独立入口：** 安全卫士始终保持独立可达；当存在待确认、被拦截或可恢复事项时，应直接高亮。

• **任务状态：** 承载正在推进的任务、已结束任务、步骤时间线、成果入口、风险与恢复信息。

• **便签协作：** 承载近期要做、后续安排、重复事项与已结束事项，是“未来安排”向“执行任务”转化的入口。

• **镜子：** 承载对话概要、日报、用户画像、习惯洞察与记忆管理，是长期理解能力的展示面。

• **安全卫士：** 承载授权、拦截、审计、影响范围、恢复点、规则边界和工作区信息，是信任与边界的统一入口。

# 3. 首页总览与全局交互

## 3.1 首页长期承载的信息

- **主焦点摘要：** 显示 Agent 当前状态、当前任务标题、下一步动作和最近更新时间，让用户一眼知道事情推进到了哪里。
- **信任摘要：** 显示当前风险级别、待授权数量、是否存在恢复点，以及成果保存位置。
- **召唤入口：** 用于承接紧急提醒、到期事项、总结结果、成功归档等低侵入提示。
- **快速操作：** 至少包括打开当前任务、查看待确认、长按中心球发起语音、回到首页。
- **全局状态：** 显示 Agent 是否暂停、是否开启主动建议、当前工作区位置。

## 3.2 中心球、轨道节点与分离窗口

- **中心球：** 支持拖拽、长按语音、返回首页。它不是装饰物，而是“Agent 始终在场”的视觉锚点。
- **模块行星：** 支持单击进入模块，必要时可拖拽后回到轨道。模块之间应被感知为“可切换的对象”，而不是一排静态标签。
- **详情面板：** 进入模块后，面板应支持左右切换当前模块的不同状态，上下切换模块本身，减少层级跳转成本。
- **分离窗口：** 任何关键详情都应支持拖出，像便签一样固定在桌面边缘，随时收起或召回，适应一边工作一边协作的场景。

## 3.3 专注模式与召唤机制

专注模式适合用户只想盯住当前任务时使用。进入后应隐藏其他行星，只保留中心球与当前内容；退出后，再把其他模块按顺序带回。

召唤机制用于 Agent 主动发起低侵入提醒。建议至少区分三档：

- **紧急：** 会影响当前任务结果、涉及授权、已逾期或异常中断，应允许用户一键进入对应对象。
- **普通：** 与近期任务、总结结果、便签巡检直接相关，可通过轻卡片或状态变化提示。
- **低优先级：** 更多用于到场感和背景提醒，不应抢焦点，也不应默认弹窗。

无论哪一档提醒，点击后都应直接落到对应模块和对应对象，而不是只打开一个空页面。

# 4. 任务状态模块

## 4.1 模块定位

任务是“已经被 Agent 接手并正在推进的工作”；待办是“未来的安排”。当便签协作中的某件事正式交给 Agent 推进后，应自动生成一个任务，并与原始待办保持关联。

任务列表建议分为“未完成”和“已结束”两组，并按最近更新时间或时间临近程度排序。已结束任务默认展示近 3 天，可切换到近 7 天与更多；点击“更多”时，建议直接打开相应成果或记录所在的文件夹。

## 4.2 预览态要承载的内容

- **未完成任务：** 至少展示任务名称、任务状态，以及当前执行到哪一步。状态命名应足够直白，例如暂停、等待授权、正在进行、失败。
- **已结束任务：** 至少展示任务名称和最终状态，例如已完成、已取消、已结束未完成。
- **首页映射：** 首页焦点区不需要把所有细节都展开，但必须让用户快速判断这个任务是否卡住、是否要我现在介入。

## 4.3 详情态要承载的内容

- **任务头部：** 包含任务名称、当前状态、来源、开始时间、最近更新时间。
- **进度时间线：** 完整列出步骤，并明确当前停留在哪一步；已完成、进行中、未开始必须有清晰区分。
- **关键上下文：** 展示本次任务用到的资料、记忆摘要、用户约束与重要前提。
- **成果区：** 展示已经生成的文件、草稿、网页、清单、模板等，并支持直接打开。
- **信任摘要：** 展示风险状态、待授权数量、是否存在恢复点、最近一次边界触发。
- **操作区：** 至少提供暂停或继续、取消、修改任务、查看安全详情、重启任务等动作。

## 4.4 建议采用的状态语言

- **进行中：** 例如正在进行、接近完成。重点展示当前步骤、剩余动作和预计产出。
- **等待中：** 例如暂停、等待授权、等待补充信息、等待上游条件。重点解释“为什么停住”与“需要谁处理”。
- **异常中：** 例如失败、阻塞、执行异常。重点展示最近一步、问题原因、可恢复点和建议处理方式。
- **已结束：** 例如已完成、已取消、已结束未完成。重点展示关键操作摘要、最终结果、产出位置和重启入口。

## 4.5 关键交互

- **任务推进顺利：** 首页焦点持续显示推进中的步骤，在接近完成时突出成果预览，方便用户立即打开或复用。
- **任务等待授权：** 任务卡与安全入口要同时高亮，并清楚说明为什么需要授权。
- **任务被阻塞：** 要直接告诉用户卡在什么条件、缺少什么信息，而不是只显示“失败”。
- **任务完成：** 除了展示成果，还应允许用户一键重启、复用为模板或跳回相关便签。

# 5. 便签协作模块

## 5.1 模块定位

便签协作不是传统待办清单，而是 Agent 异步协作的入口。它负责把“记下来的事情、周期性的事情、尚未开始但需要记住的事情”整理成可巡检、可提醒、可转交给 Agent 的结构。

待办与任务的关系要明确：**便签协作**是未来的安排，**任务状态**是正在被推进的工作。当**便签协作**中的某项内容正式进入执行时，应自动新建任务；原待办仍保留在便签协作中，作为后续安排或重复事项的来源记录。

## 5.2 分类结构与预览逻辑

- **近期要做：** 适用于快到时间、今天要做、最近几天要处理的事项；其中应包含重复事项在当天落地的实例。首页或模块预览中，至少显示事件名称、状态与时间临近程度。
- **后续安排：** 适用于已经记下但还没到处理窗口的事项。预览中，至少显示事件名称与“未到时间”的状态。
- **重复事项：** 适用于每天、每周、每月、工作日等循环发生的安排。预览中不展示某一次实例，而展示规则名称与下一次发生时间。
- **已结束：** 适用于已完成或已取消的事项。预览中只需要显示事件名称和最终状态。

## 5.3 各类事项的详情态

- **近期要做：** 详情中应包含事件名称、状态（即将到来或已逾期）、时间信息、距离开始或截止还有多久、事件类型、备注、Agent 建议和相关资料入口；核心动作包括标记完成、取消或跳过本次、编辑、打开相关资料。
- **后续安排：** 详情中应包含计划时间或截止日期、前置条件与关联资料；核心动作包括提前到近期要做、编辑、取消与打开相关资料。它与“近期要做”的区别在于：前者强调已安排但暂时不用处理，后者强调快要发生。
- **重复事项：** 详情应被理解为“规则本身”的详情页，而不是某一次实例的详情页。需要展示重复规则、下次发生时间、最近一次状态、生效范围与相关资料；核心动作包括暂停或开启重复、修改规则、取消整个重复事项、打开相关资料。
- **已结束：** 详情应展示事件名称、最终状态、原定时间、结束时间、事件类型，并提供恢复为未完成与删除记录等动作。

## 5.4 页面上的信息组织

- 顶部建议有一条摘要：今天待处理多少项、逾期多少项、重复事项今天落地多少项，其中多少项适合直接交给 Agent。
- “近期要做”默认优先显示，因为它最接近行动。
- “重复事项”展示规则本身；如果某一次实例已经进入处理窗口，就在“近期要做”里展示。
- 每条事项都尽量给出可执行建议，而不是只停留在提醒层面。

## 5.5 Agent 建议的表现方式

- **下一步建议：** 在事项卡片底部附一条清晰动作，例如先打开模板、先整理材料、先生成草稿。
- **代办转任务：** 当用户确认后，应直接生成任务并跳转到任务状态页。
- **流程化建议：** 当某类事项反复延期或频繁出现时，可提示是否整理成固定流程或模板。

# 6. 镜子模块（记忆与洞察）

## 6.1 模块定位

镜子模块承接的是 Agent 的长期理解能力，但页面表现不能像原始日志仓库，而应像“可以被用户理解和管理的认知镜像”。它至少要承担三件事：

- 替代传统聊天里的“历史承接”，让用户能看到最近一段时间的对话概要与上下文脉络。
- 生成日报或阶段总结，让用户知道今天做了什么、完成了什么、涉及了哪些关键权限或成果。
- 逐步沉淀用户画像、偏好与习惯，并允许用户查看、纠正、关闭或删除。

## 6.2 预览态与详情态

- **首页预览：** 只展示一条当前最有价值的洞察，例如“这周反复出现的 3 类工作”或“最近常在同一时间段处理周报”。
- **详情标签：** 建议按“历史概要 / 日报或阶段总结 / 用户画像 / 记忆管理”四个标签组织，避免把总结、偏好和原始记录混在一起。
- **历史概要：** 展示按对话或任务阶段整理出的摘要，而不是原文堆叠；用户可以查看概要，并跳转到本地记录位置。
- **日报与阶段总结：** 展示当天完成事项、成果、权限、异常和重要提醒，并支持按天回看。
- **用户画像：** 展示偏好、习惯、长期目标与常用工作节奏，并支持编辑、纠正、删除或关闭某类记忆。
- **近期被调用记忆：** 展示当前任务用了哪些记忆、为什么被引用，增强透明度，避免建议像“凭空出现”。

每条洞察都应附带时间范围或来源范围，让用户知道它不是凭空生成；记忆也不是只读内容，用户应能执行保留、修改、删除、忽略本次建议、关闭该类记忆等动作。

# 7. 安全卫士（信任与边界）

## 7.1 模块定位

安全卫士是仪表盘里的最后一道解释与接管层。它不应该只在出事时出现，而应始终让用户知道：当前有没有待确认操作、最近是否发生过拦截、成果会保存到哪里、如果现在停下能否恢复。

建议采用一组用户可理解的状态语言：正常、待确认、已拦截、执行异常、可恢复、已恢复。

## 7.2 预览态与详情态

- **预览态：** 至少展示当前安全状态、待确认操作数、最近一次异常和最近恢复点。
- **待确认 / 被拦截：** 详情中应展示操作名称、风险等级、目标对象、触发原因，并提供允许本次、拒绝本次、查看详情等动作。
- **授权记录：** 展示历史授权结果、时间与对象，帮助用户形成信任感与边界感。
- **审计记录：** 展示文件、网页、命令、系统动作的关键摘要，并支持按类型查看。
- **影响范围：** 展示受影响的文件、网页、应用，说明是否越出工作区、是否覆盖已有内容。
- **恢复与回滚：** 展示恢复点时间、关联任务、涉及对象，并支持恢复到该节点或先查看影响范围。
- **规则与权限：** 展示绿灯、黄灯、红灯的边界，以及工作区范围、是否允许某类动作等规则。

## 7.3 风险等级的页面表达

- **绿灯：** 适合承载低风险、可静默执行的读取与查看动作；页面记录即可，不打断当前流程。
- **黄灯：** 适合承载会修改内容、会触发下载、会改变环境的动作；页面应先解释影响范围，再给确认入口。
- **红灯：** 适合承载删除、外发、不可逆、越界或涉及身份与关键文件的动作；必须显式人工确认，并强调后果。

安全页的设计重点不是把风险抽象化，而是把风险具体化：让用户知道会改什么、会影响谁、能不能撤回。

# 8. 高价值信息球

> 用户打开仪表盘时，仪表盘意图判断当前用户可能最关注事件什么情况，然后弹出一个高价值信息球

## 展示内容：

# 9. 跨模块联动设计

## 9.1 关键联动链路

- **便签协作 → 任务状态：** 当事项进入执行时，自动生成任务，并保留原事项的来源关系，避免待办与执行割裂。
- **任务状态 → 安全卫士：** 一旦出现授权、拦截、异常或越界风险，任务页要显示安全摘要，安全页则承接详细解释。
- **镜子 → 任务状态：** 当前任务命中历史偏好或相似经验时，要在任务详情中明确展示“使用了哪些记忆”。
- **硬件感知 → 任务状态：** 当系统状态影响执行效果时，要在任务页直接标记“建议暂缓”或“适合继续”。
- **任一模块 → 首页召唤：** 遇到到期、完成、总结、异常、保存成果等节点时，应通过首页召唤卡片把用户带回当前重点。

## 9.2 提示层级建议

**•**  默认先用状态变化和轻提示卡片，不直接强弹窗。

**•**  用户有明确意图时，例如打开任务文件、复制文本、临近截止、出现错误，才提高提示强度。

**•**  涉及发送消息、改文件、执行系统动作时，必须提供确认、可撤销和可审计的信息。

**•**  弱信号，例如长时间停留、频繁切换窗口，只适合静默建议，不应直接触发强提示。

# 10. 固定快捷操作与统一导航

快捷操作属于仪表盘内部的固定规则，目标是在不依赖鼠标的情况下，快速切换模块、收起面板、进入专注模式。建议保留一套稳定、易记的默认映射：

**• F：** 进入或退出专注模式，只保留当前核心内容。

**• 1–4：** 快速切换任务状态、便签协作、镜子、硬件感知。

**• 5：** 打开安全卫士。

**• ？：** 打开快捷操作帮助。

**•Esc：** 关闭当前面板或退出语音界面。

**• 方向键：** 在当前模块状态和不同模块之间快速导航。

## 10.1 统一空态、等待态、异常态与完成态

**• 空态：** 要告诉用户“现在没有内容”，并给一个明确入口，而不是只留下空白页面。

**• 等待授权：** 要说明为什么需要授权、授权后会发生什么，而不是只报“权限不足”。

**• 异常态：** 要说明卡在哪、已完成什么、还可怎么恢复，而不是只显示失败结果。

**• 完成态：** 要展示关键成果、保存位置，以及是否可重启或复用，而不是任务结束后直接消失。

# 11. 结论

完善后的仪表盘应把“看见状态、接管任务、理解建议、授权与恢复”收束成同一套体验：首页负责判断轻重缓急，详情页负责解释和操作，安全卫士负责兜底，镜子与硬件感知则为当前建议提供背景与依据。这样它才不是一个静态面板，而是桌面端 Agent 的稳定协作界面。

# 12.原型的交互设计说明

### 主页面 — 意识聚焦场

- 中心球：长按（0.65s）唤起语音场；可拖拽，松手弹回；下方显示 Agent 循环消息
- 三层轨道球：绿色任务球（焦点层）、紫色便签/镜子球（候选层）、蓝色感知球（背景层）；点击打开对应面板；可拖拽到轨道任意位置
- 外围关注标签：多 个动态语义碎片，可拖拽移动位置，hover 显示 × 关闭，关闭后底部出现「已隐藏 · 撤销」提示
- 召唤球：每隔约 12s 从暗处浮现一个高价值信息球，可拖拽，hover 暂停倒计时，点击打开面板
- 按 F 进入专注模式：所有外围内容淡出，只剩中心球

### 语音场（长按中心球触发）

- 4 段式：待唤起 → 正在听（关键词回声漂浮）→ 正在理解（粒子向心收束 + 意图碎片）→ 确认执行（摘要 + 操作按钮）→ 执行中（进度弧 + 步骤文字）
- 底部有「你现在可以说」的意图建议词，点击直接触发

### 模块面板（点击轨道球打开）

- 顶部 tab 切换模块，状态时间线/胶囊切换子状态
- 内容区展示任务进度、便签、镜子洞察等
- 可自由缩放（左/右/底/角拖拽）
- 长按面板内容区 0.8s 可拖出为独立浮窗

### 独立浮窗（长按拖出）

- 可自由拖动位置，8 方向自由缩放
- 多窗口支持，点击置顶
