# CialloClaw 开发统一规范（修订版 v18）

## 1. 文档定位

本文档用于冻结 CialloClaw 在正式开发阶段必须统一遵守的工程规则。  
它不是架构总览，也不是协议真源，更不是模块时序说明；它回答的是：

- 哪些对象、术语、状态、接口和目录必须统一；
- 哪些设计原则已经从架构层下沉为开发铁律；
- 哪些做法允许，哪些做法禁止；
- 哪些内容在编码前必须先冻结，不能边写边改。

---

## 2. 开发总原则

### 2.1 长期目标

- 保持桌面现场优先，而不是聊天窗口优先。
- 保持前后端分离，而不是把能力散落在前端页面里。
- 保持统一协议、统一对象、统一状态和统一交付。
- 保持主链路优先，先保证可演示、可恢复、可审计，再做扩展能力。

### 2.1.1 架构设计原则补充

除长期目标外，开发侧还必须遵守以下架构设计原则，这些原则来自上位架构方案，进入开发阶段后视为工程铁律的一部分：

1. **桌面现场优先**：优先围绕当前页面、选中文本、拖入文件、错误信息、系统状态承接需求，而不是先让用户进入聊天页补上下文。
2. **轻量承接优先**：轻量对话、意图确认与即时结果优先由悬浮球附近的气泡和轻量输入区承接；只有需要完整状态或持续任务时才进入仪表盘。
3. **先提示、再确认、后执行**：改文件、发消息、系统动作、工作区外写入等场景必须经过风险评估、授权确认、审计记录和恢复点保护。
4. **前馈引导先于自由执行**：每次关键 LLM 调用前，都必须由 Context Manager、Memory、Skill、Prompt Composition、AGENTS 规则、架构约束、LSP 与 Blueprint 共同收敛输入。
5. **事件驱动、可恢复**：系统执行链路采用“观察—规划—执行—校验—持久化—恢复”的闭环，关键步骤进入事件流，并在关键边界保存 checkpoint。
6. **反馈闭环内建**：Linter / CI、Test Harness、Agent Review、Hooks、Trace / Eval、Doom Loop 检测、Entropy Cleanup 与 Human-in-the-loop 是 Harness 标准组成，而非附加功能。
7. **记忆与运行态分离**：长期记忆、阶段总结和画像与任务运行态、审计、恢复点严格分层；记忆检索只能通过 Memory 内核接入，不得直接侵入运行态状态机。
8. **统一出口优先**：正式结果统一通过 `delivery_result / artifact / citation` 体系发布，禁止工具、worker、SubAgent、插件绕过交付内核直接对前端返回临时结构。
9. **抽象先于平台细节**：文件系统、路径、进程、容器执行、屏幕采集、通知、快捷键、剪贴板等能力必须先定义接口，再分别实现平台适配。
10. **前后端严格解耦**：前端只通过 JSON-RPC 调用后端；后端不感知 Tauri、React、页面路由与组件树。
11. **主链路优先**：所有开发与联调必须优先服务 P0 演示链路，禁止绕过主链路先做外围能力。
12. **模型接入标准化**：模型统一通过官方 SDK 接入层与配置入口进入，不得散落在业务代码中。

### 2.1.2 设计原则与开发总原则的对应关系

- “桌面现场优先”“轻量承接优先”要求前端交互、协议入口和交付方式始终围绕当前任务现场设计，不得把聊天式长对话默认化。
- “先提示、再确认、后执行”“事件驱动、可恢复”“反馈闭环内建”要求高风险动作、长任务和失败路径全部进入风险、审计、恢复、Trace 和评估链路。
- “前馈引导先于自由执行”“记忆与运行态分离”要求 Context Manager、Memory、Skill、Blueprint、Prompt 资产和约束规则先于模型调用生效，且不得让记忆逻辑直接侵入运行态状态机。
- “统一出口优先”“前后端严格解耦”“抽象先于平台细节”“模型接入标准化”要求协议、交付、平台、模型接入和对象模型全都通过统一边界进入实现。

### 2.2 协作方式

- 所有新增能力优先映射到现有主链路，不单独开平行链路。
- 所有新增字段、方法、状态、对象名、目录名必须先检查是否已被统一冻结。
- 所有工程实现都要优先写清输入、输出、状态和边界，再开始编码。

### 2.3 铁律

1. 不允许前端直接调用 worker、LSP、Playwright、数据库或模型 SDK。
2. 不允许后端跳过 JSON-RPC 契约直接约定“隐式接口”。
3. 不允许新增一个“差不多但名字不同”的对象。
4. 不允许状态枚举、错误码、方法名在多个地方各写一份。
5. 不允许把轻量 UI 状态当成正式业务状态。
6. 不允许为了快先写平台专属逻辑，再说以后抽象。
7. 不允许为了赶工绕过风险评估、授权确认、审计与恢复点。
8. 不允许新功能打穿 `task` 主链路和统一交付出口。

### 2.4 设计原则到工程执行的落地要求

- 所有需求评审、模块设计、协议新增、表结构新增和 AI 生成代码评审，都必须先检查是否违反上面的架构设计原则。
- 若某实现方案与“桌面现场优先、轻量承接优先、统一出口优先、抽象先于平台细节、前后端严格解耦”冲突，则即便功能可跑通，也不得直接进入主干。
- 若某功能无法同时满足“安全治理、恢复能力、Trace 可观测、对象统一出口”四项要求，则最多只能作为实验分支，不得进入正式里程碑。

## 5. 统一命名与核心对象规范

### 5.1 统一命名规则

- 类型名：`PascalCase`
- 字段名：`snake_case`（数据库、协议字段）
- 前端 TS 字段：与协议字段一一映射，不自行改名
- 方法名：JSON-RPC 使用 `agent.domain.action`
- 事件名：使用 `dot.case`
- 工具名：使用 `snake_case`

### 5.2 核心保留词

以下词汇属于系统核心词，不能随意换名：

- `session`
- `task`
- `task_step`
- `run`
- `step`
- `event`
- `tool_call`
- `delivery_result`
- `artifact`
- `citation`
- `approval_request`
- `authorization_record`
- `audit_record`
- `recovery_point`
- `memory_summary`
- `memory_candidate`
- `retrieval_hit`
- `trace_record`
- `eval_snapshot`
- `skill_manifest`
- `blueprint_definition`
- `prompt_template_version`

### 5.3 核心词简要说明

- `session`：一组相关协作任务的会话容器。
- `task`：对外正式任务主对象，是前端和交付层的锚点。
- `task_step`：面向用户展示的任务阶段时间线对象。
- `run`：后端执行实例，是 task 的执行兼容对象。
- `step`：run 内部的细粒度执行步骤。
- `event`：任务和执行过程中的结构化事件。
- `tool_call`：工具、worker、sidecar 输出的统一封装对象。
- `delivery_result`：正式结果出口对象。
- `artifact`：正式产物对象，如文档、文件、截图、结构化结果。
- `approval_request`：待授权动作对象。
- `authorization_record`：用户授权决策记录。
- `audit_record`：关键动作审计记录。
- `recovery_point`：恢复与回滚锚点。
- `memory_summary`：正式长期记忆对象。
- `memory_candidate`：候选记忆对象。
- `retrieval_hit`：一次检索中的命中记录。
- `trace_record`：一次执行链路中的追踪记录。
- `eval_snapshot`：评估快照。
- `skill_manifest`：Skill 配置与版本元数据。
- `blueprint_definition`：蓝图定义。
- `prompt_template_version`：Prompt 模板版本对象。

### 5.4 原子功能命名映射规则

原子功能若进入系统实现，必须先映射到以上核心对象与功能域，不能直接以 PRD 名称落代码。

例如：

- “任务巡检” → `todo_item / recurring_rule / task`
- “镜子” → `memory_summary / retrieval_hit`
- “安全卫士” → `approval_request / audit_record / recovery_point`
- “结果承接” → `delivery_result / artifact`

## 6. 状态真源与状态约束

### 6.1 正式状态真源

正式状态只能来自以下位置：

- 协议文档定义的枚举
- 数据设计文档定义的主表字段
- 后端编排层状态机

### 6.2 前端局部状态

前端允许存在局部状态，但这些状态不是真正业务状态真源，例如：

- 悬浮球态
- 气泡生命周期
- 输入框草稿态
- 悬停推荐可见态
- 设置页面未保存态

### 6.3 状态约束

- 前端局部状态不能覆盖正式业务状态。
- `task.status` 不能直接等于 `run.status`。
- `TodoItem.status` 不能直接映射为 `Task.status`。
- 授权中的等待态必须有正式对象锚点。
- 失败、熔断、人工升级和回滚必须有结构化状态体现。

## 7. JSON-RPC 真源规则

### 7.1 方法集合说明

JSON-RPC 方法集合是前后端唯一正式调用入口。  
它回答的是：前端能调什么、参数是什么、结果是什么、错误怎么返回。  
所有正式联调能力必须先进入协议方法集合，不能在实现中“临时约定”。

### 7.2 状态约束说明

协议中的状态定义属于系统冻结项。  
这意味着：

- 状态枚举不能由前端页面自行扩展
- 不能由 worker 自创一套结果状态
- 不能通过文案推测真实状态

### 7.3 方法命名规则

统一使用：

`agent.domain.action`

例如：

- `agent.input.submit`
- `agent.task.start`
- `agent.task.confirm`
- `agent.task.cancel`
- `agent.task.query`
- `agent.notepad.inspect`
- `agent.notepad.convert_to_task`
- `agent.recommendation.get`
- `agent.recommendation.feedback.submit`
- `agent.settings.get`
- `agent.settings.update`

### 7.4 事件命名规则

统一使用 `dot.case`：

- `task.updated`
- `delivery.ready`
- `approval.pending`
- `plugin.updated`

### 7.5 协议禁止事项

- 不允许新增未登记方法。
- 不允许把通知当作隐藏命令通道。
- 不允许一个能力同时保留两套名字相近的方法。
- 不允许返回未登记字段。
- 不允许错误码和错误文案失配。

## 8. 错误码与错误处理规范

### 8.1 错误码体系

统一使用 `100xxxx` 整型错误码。  
不同域必须预留段，例如：

- `1001xxx`：输入与参数错误
- `1002xxx`：任务与状态错误
- `1003xxx`：授权与安全错误
- `1004xxx`：工具与 worker 错误
- `1005xxx`：平台与文件系统错误
- `1006xxx`：模型与推理错误

### 8.2 错误结构

必须包含：

- `code`
- `message`
- `details`

### 8.3 错误处理原则

- 后端返回正式错误对象，不透传栈信息。
- 前端按错误码做可恢复提示。
- 高风险失败必须引导到安全详情或恢复点。
- 错误必须可进入 Trace / Eval。

## 9. 原子功能落地规则

### 9.1 原子功能必须映射

任何原子功能在落代码前，都必须先回答：

1. 进入哪个功能域？
2. 前端由哪个模块承接？
3. 后端由哪个模块编排？
4. 影响哪些核心对象？
5. 是否新增协议方法？
6. 是否新增表、字段、索引？
7. 是否进入风险与恢复链？

### 9.2 原子功能禁止事项

- 不允许只做 UI 外壳而没有后端对象。
- 不允许只做后端能力而没有正式交付出口。
- 不允许为了 demo 写绕开主链路的旁路。
- 不允许原子功能直接打破已冻结对象模型。

## 10. 数据主模型与表设计对齐规则

### 10.1 数据主模型

以下主模型必须冻结：

- `Session`
- `Task`
- `TaskStep`
- `Run`
- `Step`
- `Event`
- `ToolCall`
- `DeliveryResult`
- `Artifact`
- `ApprovalRequest`
- `AuthorizationRecord`
- `AuditRecord`
- `RecoveryPoint`
- `MemorySummary`
- `MemoryCandidate`
- `RetrievalHit`

### 10.2 数据变更要求

- 新表必须说明用途、主键、外键、索引、读写方、生命周期。
- 新字段必须说明是否进入协议、是否被查询、是否需要索引。
- DDL 变更必须同步回写数据设计文档。
- 不能只贴 SQL，不写说明。

### 10.3 数据分层要求

- 运行态、记忆、前馈配置、Trace/Eval 必须分层。
- `task` 与 `run` 双层模型不能混写。
- `todo_items` 与 `tasks` 必须分层。
- `trace_records` 不能拿来代替正式业务对象。

## 11. 跨平台抽象与执行隔离规范

### 11.1 抽象原则

业务逻辑不能直接依赖平台实现。  
必须先定义接口，再实现 Windows 版本；未来 Linux/macOS 通过相同接口接入。

### 11.2 必须抽象的接口

- `FileSystemAdapter`
- `PathPolicy`
- `OSCapabilityAdapter`
- `ExecutionBackendAdapter`
- `StorageAdapter`

### 11.3 Windows 当前唯一落地原则

当前只实现 Windows，但不允许在业务代码中写死未来无法迁移的路径和行为。

### 11.4 平台专属代码要求

- 平台专属逻辑只能放在平台适配层。
- 业务层不能直接调用 Win32、PowerShell、Shell API。
- 平台路径不能散落在业务代码中。

### 11.5 路径与文件系统约束

- 所有路径必须经过 PathPolicy。
- 所有文件系统写入必须经过 FileSystemAdapter。
- Workspace 外写入必须进入风险链。

### 11.6 执行隔离规则

执行隔离分为：

1. 宿主治理层
2. 外部执行后端层

高风险动作必须经过风险评估、授权确认、恢复点和审计后，才能进入 Docker Sandbox 或其他执行后端。

### 11.7 部署原则

- 当前仅交付 Windows 安装包与 Windows 运行闭环。
- 以前后端一体分发、内部多进程解耦为原则，不拆分为公网服务。
- Go local service 必须作为应用 sidecar 随桌面端一起安装与启动。
- Node Playwright sidecar、媒体 worker、LSP worker 必须保持独立进程与独立升级能力。
- 升级链路必须使用签名校验，禁止未签名二进制直接替换。
- 部署策略必须优先保证主链路可运行，再考虑扩展能力按需启用。
- 平台专属能力必须通过抽象接口接入，禁止在部署脚本或业务实现中写死未来跨平台不可迁移的主逻辑。

### 11.8 部署与分发规则

开发与发布必须遵守以下分发约束：

- 桌面壳通过 Tauri Windows 安装包分发。
- 后端 Harness 通过随桌面壳分发的 Go sidecar 提供。
- 浏览器自动化、媒体处理、LSP 必须以独立 worker / sidecar 二进制或 Node runtime 组件方式分发。
- 数据目录必须按 Workspace、SQLite、Artifact、Stronghold 分目录管理。
- 升级必须支持安装包升级、worker 独立升级，以及失败时回滚到上一个稳定版本。
- 任何新能力若改变安装包结构、目录布局、sidecar / worker 生命周期，都必须先补文档并评估是否影响主链路升级与回滚。

### 11.9 部署实现禁止事项

以下情况直接视为违反统一规范：

- 把 Go service、worker 或 sidecar 直接改造成公网常驻服务以替代本地闭环。
- 在业务实现中写死平台安装路径、升级路径、临时目录或日志目录。
- 未经过签名校验直接替换二进制或执行更新。
- 把 worker 和主应用编译成无法独立升级的硬耦合单体。
- 把部署、升级、回滚逻辑散落在多个业务模块中而不经过统一发布与启动策略。

## 12. LLM / AI 接入与扩展能力规范

### 12.1 模型接入规则

- 模型统一通过官方 SDK 接入层进入。
- 不允许在业务代码中直接 new 客户端。
- 模型 ID、提供商、路由策略必须配置化。

### 12.2 前馈资产要求

以下资产必须版本化：

- Skill
- Blueprint
- Prompt Template
- AGENTS 规则片段
- 架构约束规则

### 12.3 LSP 与代码语义能力要求

- LSP 属于前馈引导能力，不只是普通工具。
- 定位、诊断、hover、rename 等能力必须通过统一接口进入。
- LSP 结果必须可进入 Context Manager 和 Trace。

### 12.4 插件与扩展能力要求

- 插件必须通过 Go service 编排。
- Skill / 插件 / 感知包 / 模型路由都必须可追踪版本、来源和权限。
- 扩展能力不能改变核心协议对象与主数据模型。

---

## 13. 测试、Trace、评估与审查规范

### 13.1 Linter / CI 优先原则

Linter / CI 是最快、最便宜、最确定的反馈源。  
其错误优先于模型自解释。

### 13.2 测试要求

- 核心链路必须有结构化测试。
- 关键状态迁移必须可验证。
- 回归问题必须有测试锚点。
- 高风险链路必须验证恢复点与审计。

### 13.3 Trace / Eval 强制记录项

以下内容必须结构化记录：

- 每次关键 LLM 调用输入 / 输出摘要
- latency / cost
- rule hits
- Skill 命中
- Blueprint 选择
- LSP 诊断结果
- Memory 命中
- Hook 拦截
- 审查结论
- Doom Loop 命中

### 13.4 审查要求

- 规则无法覆盖的语义问题使用 Agent Review。
- 审查结果必须可回流状态机、升级流程或失败报告。
- 高风险输出不能只做前端提醒，必须有正式对象。

### 13.5 数据变更要求

- 新表必须说明：用途、主键、外键、索引、读写方、生命周期。
- 新字段必须说明：是否进入协议、是否进入查询条件、是否需要索引。
- DDL 变更必须同步回写数据设计文档。

### 13.6 可观测指标与埋点要求

架构文档第十章中的“可观测指标”在开发阶段必须落为统一埋点与指标口径。以下指标应做成一等指标，不得只停留在口头约定：

- 每次调用输入 / 输出 Token
- 各上下文块占比
- Skill 命中率
- Blueprint 复用率
- LSP 诊断命中率
- 审查通过率
- Hook 拦截次数
- RAG 命中率
- 记忆候选过滤命中率
- Doom Loop 命中率
- 熔断触发率
- worker / sidecar / plugin 失败率

补充要求：

- 指标名、事件名、trace 字段名必须能稳定映射到 `/packages/protocol` 或 Trace / Eval 数据模型。
- 新增 worker、plugin、模型路由或前馈策略时，必须同时说明其指标如何进入观测面。
- 不允许只在日志字符串里输出关键指标，而没有结构化追踪字段。

### 13.7 成本控制策略补充

开发实现必须遵守以下成本控制策略，不得把成本治理留到后期再补：

- 上下文预算必须前置分配，而不是等超限后再被动截断。
- 历史上下文优先采用摘要继承，不直接无限回灌原始对话。
- 工具结果默认回填摘要，不直接把大块原始输出无条件塞回模型。
- 小模型 / 规则预处理，大模型负责核心推理与生成。
- 长任务必须支持阶段性压缩与阶段总结。
- 成本超阈值时必须支持熔断、降级或切换较低成本路径。

落地要求：

- 成本治理必须体现在编排层、Trace / Eval、设置项与安全 / 预算策略中。
- 任何新功能若显著增加 token、上下文长度、工具调用次数或媒体处理成本，必须在设计评审阶段说明其成本上界和降级方案。
- 不允许把“预算降级”只做成 UI 文案而不进入真实执行策略。

## 14. 风险治理与恢复链规范

### 14.1 风险等级

统一使用：

- `green`
- `yellow`
- `red`

### 14.2 高风险动作定义

包括但不限于：

- 工作区外写入
- 删除 / 覆盖文件
- 外发消息
- 执行命令
- 浏览器自动化提交
- 系统级动作

### 14.3 必须经过的链路

1. 风险评估
2. 影响范围生成
3. 授权确认
4. 审计记录
5. 恢复点
6. 受控执行
7. 结果回写

### 14.4 恢复要求

- 失败或中断必须可追溯。
- 恢复点必须在动作前创建。
- 回滚结果必须进入正式对象链。

---

## 15. 交付出口与前端承接规范

### 15.1 正式交付出口

正式结果只能通过：

- `delivery_result`
- `artifact`
- `citation`

### 15.2 轻量承接

轻量提示与近场反馈允许使用：

- `BubbleMessage`
- 局部 UI 状态

但它们不是正式业务对象出口。

### 15.3 结果分流原则

- 短结果 → 气泡
- 长文本 → 工作区文档
- 结构化结果 → 结果页 / 浏览器
- 文件产物 → 文件定位
- 连续任务 → 任务详情
- 失败 / 中断 → 结构化说明 + 恢复入口

---

## 16. 禁止事项总表

1. 不允许前端直调数据库、worker、模型或工具。
2. 不允许绕过协议真源。
3. 不允许自创对象名、状态名、方法名。
4. 不允许未授权执行高风险动作。
5. 不允许平台专属逻辑污染业务层。
6. 不允许把记忆、Trace、前馈配置混入主业务表。
7. 不允许未审计的关键动作进入主链路。
8. 不允许未登记的扩展能力进入主干。
9. 不允许只改代码不改文档真源。

当架构文档、协议文档、数据设计文档、模块设计文档与本规范发生冲突时：

- **表结构与索引真源** 以数据设计文档为准；
- **模块职责与流程说明** 以模块详细设计文档为准；
- **系统边界与技术取舍** 以架构总览文档为准。
