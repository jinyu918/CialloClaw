# CialloClaw 分工安排和优先级划分（修订版 v12）

## 1. 文档目的

本文档用于在以下上位文档之下，给出一份可直接执行的协作与排期方案：

1. 架构总览文档
2. 开发统一规范（修订版 v16）
3. 协议设计文档（v4）
4. 数据设计文档（v4）
5. 模块详细设计文档

本文档回答五类问题：

1. 当前阶段什么能力优先级最高
2. 五个人分别负责什么模块与原子功能
3. 谁依赖谁、交付给谁
4. 各阶段里程碑怎么拆
5. 如何避免跨边界和重复建设

---

## 2. 协作口径

### 2.1 对外主对象统一为 task-centric

- 前端、工作台、正式交付统一使用 `task`
- 后端执行层使用 `run / step / event / tool_call`
- `task_id` 与 `run_id` 必须稳定映射

### 2.2 正式协议统一为 agent 分组

正式方法统一使用 `agent.xxx.xxx`。

### 2.3 当前唯一主链路

**语音 / 悬停输入 / 文本选中 / 文件拖拽 → 悬浮球承接 → 意图确认 → 创建或更新 task → Go service 编排工具链 → 风险动作挂起等待授权 → 正式交付 → 仪表盘展示 task / artifact / audit / recovery_point → 完成一次记忆命中或记忆摘要回写。**

### 2.4 文档拆分口径

当前保持 5 份执行文档视角：

- 架构：讲边界和链路
- 规范：讲规则和铁律
- 协议：讲方法、状态、错误码
- 数据：讲表、字段、索引、约束
- 分工：讲谁做什么、先做什么

当前不继续细拆；若内容难懂，优先补说明和示例，而不是再拆一份小文档。

## 3. 优先级划分方法

### 3.1 判断规则

依次判断：

1. 是否影响统一项冻结
2. 是否属于唯一主链路关键节点
3. 是否是主链路可持续运行的底座
4. 是否主要是体验增强
5. 是否主要是生态化扩展

### 3.2 优先级口径

- **P0**：统一项 + 主链路必需能力
- **P1**：长期可用性、治理、恢复、可维护
- **P2**：体验与智能增强
- **P3**：生态化与规模化扩展

## 6. 模块详细设计与联调产物分配

### 6.1 时序图与状态图责任分配

- **1号**：负责近场交互相关时序与状态图的实现对齐，包括：
  - 长按语音协作
  - 悬停轻量承接
  - 协作机会承接与意图确认
  - 悬浮球状态图
  - 气泡生命周期状态图
  - 语音承接状态图
  - 意图确认状态图
- **2号**：负责工作台与结果承接相关时序：
  - 双击打开仪表盘
  - 托盘右键打开控制面板
  - 任务完成后的结果分发与交付
  - 前端任务状态图联调验收
- **3号**：负责前端平台集成、订阅桥接和状态同步说明，保证图中的窗口、托盘、文件、拖拽、RPC 行为可落地。
- **4号**：负责主链路时序：
  - 文本选中 / 文件拖拽后的意图确认与执行
  - 启动初始化与恢复
  - task / run / delivery_result 主链路一致性
- **5号**：负责治理与能力相关时序：
  - 高风险执行、授权、回滚
  - 记忆写入与检索
  - 插件执行与仪表盘展示
  - Trace / Eval / Doom Loop / 恢复点联调

### 6.2 设计补充项对应到优先级

- `平台与跨平台原则`、`平台适配层设计`：归入 **M0 + P0 底座冻结项**。
- `前馈决策架构`：归入 **4号 / 5号 的 P0 后端底座**，先做最小闭环，P1 再补版本化与命中记录。
- `反馈闭环架构`：归入 **5号 的 P1**，其中 Trace 最小记录在 P0 先落地。
- `关键时序图与状态图`：归入 **M1 联调验收物**，没有对应时序和状态联调通过，不算主链路完成。
- `后端治理与反馈层模块`：风险、授权、审计、恢复点属于 P0；Review / Hooks / Eval / Doom Loop / HITL 属于 P1。
- `后端能力接入层模块`：模型接入、工具执行、RAG 骨架属于 P0；LSP、Playwright、OCR、媒体能力完整化属于 P1 / P2。


## 7. 依赖关系

### 6.1 正确顺序

1. 4号 + 5号 先冻结协议、模型、错误码、表结构、边界
2. 3号 再把协议收成前端 client / ViewModel / 平台能力
3. 1号 + 2号 再把交互和工作台接上真实状态
4. 全员围绕唯一主链路联调
5. 最后再开 P1 / P2 / P3

### 6.2 不允许的顺序

- 1号 / 2号 先写页面再倒逼协议
- 3号 先发明本地 ViewModel 再补主对象
- 4号 / 5号 各维护一套 schema
- worker 先输出临时 JSON 再补正式对象

## 9. 文档与回写职责

### 8.1 谁回写什么

- 架构边界变化：4号 + 5号 主责，3号复核前端影响
- 协议变化：4号主责，3号和5号共同复核
- 数据表变化：5号主责，4号复核主链路影响
- 分工和优先级变化：负责人发起，所有人评审
- 原子功能新增：先补映射，再排优先级，再进入实现

### 8.2 为什么不继续拆分工文档

当前分工文档只回答“谁做什么、先做什么、怎么协作”。  
更细的接口、表、状态、异常都应回写到协议 / 数据 / 模块文档，而不是继续拆出更多协作文档。

## 10. 新手最容易踩的坑

1. 把分工当优先级
2. 前端三个人各写一套状态模型
3. 后端两个人各维护一套协议
4. 把桌面宿主能力直接写进页面组件
5. 继续沿用 run-first 前端口径
6. AI 生成代码不整理就提交
7. 把原子功能直接等价成一个“按钮功能”，不映射到对象、协议、状态和数据
